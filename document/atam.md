# 促销活动管理系统 —— ATAM 架构评估

## 1. 评估概述

### 1.1 评估目标

本文档运用 ATAM（Architecture Tradeoff Analysis Method，架构权衡分析法）对本系统的架构设计决策进行系统化评估。评估聚焦于识别架构中的**敏感点**、**权衡点**、**有风险决策**和**无风险决策**，验证架构是否满足效用树中定义的质量属性需求。

### 1.2 评估范围

评估覆盖以下架构设计决策：

| 决策编号 | 决策内容 |
|:---|:---|
| AD-01 | DDD分层 + 事件驱动的组合架构风格 |
| AD-02 | Transition Table 状态机引擎 |
| AD-03 | 独立双状态机联动校验器 |
| AD-04 | 发布-订阅模式事件总线（内存同步） |
| AD-05 | 前后端四层分层 |
| AD-06 | Pinia 集中式前端状态管理 |
| AD-07 | 路由+组件+操作三层权限控制 |

### 1.3 评估维度

基于效用树的四个质量属性维度：

| 质量属性 | 高优先级场景 |
|:---|:---|
| 可用性 | US-01（3次点击完成创建）、US-02（按钮按状态显隐） |
| 性能 | PF-01（首屏≤3秒）、PF-02（API≤500ms） |
| 安全性 | SE-01（拦截率100%）、SE-02（越权0%）、SE-03（密码加密）、SE-04（外部只读） |
| 可修改性 | MO-01（新增状态≤3处改动）、MO-02（新增审核节点≤3处改动） |

## 2. 敏感点分析

> **敏感点**：架构中对某个质量属性高度敏感的设计决策——该决策的微小变化会显著影响对应质量属性的满足程度。

### 2.1 敏感点 S1：Transition Table 的数据结构

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-02（状态机引擎） |
| **影响的属性** | 可修改性、可靠性 |
| **敏感原因** | Transition Table 的存储形式直接决定新增状态的修改范围和校验性能 |

**分析**：当前采用静态 `Map<当前状态, Map<事件类型, 目标状态>>` 结构。每新增一种活动状态，只需在Map初始化代码中添加一行映射规则。如果改用硬编码的if-else链来实现状态流转，新增状态需要在所有相关分支中添加判断逻辑，修改点极多。

**敏感度**：如果从Table驱动改为硬编码条件判断，可修改性场景MO-01/MO-02的"改动≤3处"目标将无法满足——每个涉及该状态的操作都要修改对应的if-else分支，可能涉及10处以上。

### 2.2 敏感点 S2：事件总线的同步/异步选择

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-04（事件总线） |
| **影响的属性** | 性能、可靠性 |
| **敏感原因** | 同步/异步的选择直接影响API响应时间和数据一致性保证方式 |

**分析**：当前采用内存同步模型，事件处理器在发布者线程中同步执行。同步模型的优势是事件发布和状态持久化天然在同一事务边界内，一致性强；代价是事件处理耗时直接累加到API响应时间。异步模型则相反——API响应更快但需要引入最终一致性机制。

**敏感度**：当前状态机处理逻辑是查Map+设值，耗时极短（<1ms），同步模型暂不影响性能。但一旦事件处理器逻辑变重（如增加外部服务调用），同步模型会直接影响PF-02（API≤500ms）的达标。

### 2.3 敏感点 S3：领域边界的划分粒度

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-01（DDD分层） |
| **影响的属性** | 可修改性、可用性 |
| **敏感原因** | 领域划分的粗细直接影响模块独立变更的能力和跨域协调的复杂程度 |

**分析**：当前划分为5个子领域（活动、审核、SKU、事件、用户），每个领域1~3个核心类。如果合并活动域和审核域，状态机将耦合在一起，MO-01/MO-02的独立变更目标无法满足。如果进一步拆分（如将折扣规则独立），跨域调用次数会增加。

**敏感度**：5领域的粒度在当前业务规模下是恰当的平衡点——独立的领域保障了各自演进，唯一的跨域协调（活动↔审核）通过独立的联动校验器完成，没有相互侵入。

### 2.4 敏感点 S4：权限控制粒度

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-07（三层权限控制） |
| **影响的属性** | 安全性、可用性 |
| **敏感原因** | 权限粒度决定了误操作防护的精细程度 |

**分析**：当前三层权限中，操作层按活动/审核状态组合精确控制每个按钮的可见性。如果去掉操作层只保留路由层，用户会看到不可用的按钮（如对已过期的活动显示"编辑"），US-02（按钮按状态显隐）无法满足。

**敏感度**：操作层权限直接关系到US-02和SE-02两个高优先级质量场景。

## 3. 权衡点分析

> **权衡点**：同时影响多个质量属性且方向相反的决策——提升一个属性会降低另一个。

### 3.1 权衡点 T1：分层数量 vs 简单性

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-01（DDD四层）、AD-05（前端四层） |
| **提升的属性** | 可修改性、可测试性 |
| **降低的属性** | 简单性、性能 |
| **权衡分析** | 四层分层将关注点严格分离，领域逻辑独立可测，基础设施可替换。代价是简单CRUD操作（如SKU增删改查）也要经过多层传递，调用链变长 |

**不同操作类型的层次开销对比**：

| 操作类型 | 实际执行层数 | 层次开销评估 |
|:---|:---|:---|
| SKU创建（简单CRUD） | 约3层 | 开销低，领域层较薄 |
| 审核通过（复杂流转） | 约6层 | 开销合理，每层有明确职责 |

**结论**：分层对复杂业务（活动审核流转）收益显著；对简单CRUD（SKU管理）存在一定的过度设计。但维持架构一致性带来的长期维护收益大于简单操作的层次开销。

### 3.2 权衡点 T2：事件驱动 vs 简单性

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-01（事件驱动）、AD-04（事件总线） |
| **提升的属性** | 可追溯性、可扩展性 |
| **降低的属性** | 简单性、性能 |
| **权衡分析** | 每个操作都产生事件并持久化到event_log表，提供完整的审计追溯和新业务规则的热插拔能力。代价是每次操作多一次日志写入，事件类型数从最简的5种增加到11种 |

**结论**：事件日志带来的可追溯性收益远大于额外的数据库写入开销。日志写入与业务操作在同一事务中完成，不引入额外的一致性问题。

### 3.3 权衡点 T3：集中式状态管理 vs 组件自治

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-06（Pinia集中式状态管理） |
| **提升的属性** | 跨组件状态共享、可维护性 |
| **降低的属性** | 简单性、Store代码量 |
| **权衡分析** | Pinia将活动/审核/SKU状态集中存储，任意组件都可读取最新状态，避免props多层传递。代价是Store需要维护更多响应式状态，比组件内自行管理的代码量更大 |

**场景对比**：

| 场景 | 集中式（Pinia） | 分散式（组件内管理） |
|:---|:---|:---|
| 审核操作后刷新活动列表 | Store自动响应，无需手动通知 | 需要emit事件层层传递 |
| 活动详情页状态判断 | Store统一computed计算 | 每个组件重复计算 |
| 调试追踪 | Vue DevTools可查看完整状态 | 状态分散在各组件中 |

**结论**：在6个页面共享活动/审核状态的场景下，集中式管理的收益大于代价。

### 3.4 权衡点 T4：双状态机独立性 vs 协调复杂度

| 属性 | 说明 |
|:---|:---|
| **关联决策** | AD-02（独立状态机）、AD-03（联动校验器） |
| **提升的属性** | 可修改性、可测试性 |
| **降低的属性** | 简单性 |
| **权衡分析** | 将活动状态机和审核状态机实现为两个独立引擎，通过第三个组件（联动校验器）协调。每个状态机可独立测试和修改。代价是引入了额外的协调层 |

**方案对比**：

| 方案 | 可修改性 | 可测试性 | 简单性 |
|:---|:---:|:---:|:---:|
| 当前：两个独立引擎 + 联动校验器 | 高 | 高 | 中 |
| 替代：一个大状态机统一管理 | 低（一处改全局） | 低（组合爆炸） | 高 |
| 替代：硬编码if-else | 低 | 低 | 高 |

**结论**：独立引擎+联动校验器将 O(n×m) 的状态组合问题降为 O(n+m) 的独立管理+少量联动规则，引入的复杂度是必要的。

## 4. 有风险决策

> **有风险决策**：在当前实现条件下可能无法完全满足质量属性目标的决策，需要持续关注。

### 4.1 风险 R1：内存同步事件总线

| 属性 | 说明 |
|:---|:---|
| **风险等级** | 🟡 中等 |
| **影响的属性** | 性能（PF-03 50并发） |
| **风险描述** | EventBus基于内存ConcurrentHashMap实现，事件处理器在发布者线程中同步执行。一旦处理器逻辑变重（如增加外部服务调用、复杂规则引擎），API响应时间会直接受影响 |
| **触发条件** | 事件处理器逻辑变重；或并发超过50 |
| **缓解措施** | 1. 当前处理器逻辑极轻（查Map+设值，<1ms），暂不构成瓶颈<br/>2. 架构已预留异步消息队列扩展点（Pub-Sub接口与具体实现解耦）<br/>3. 如需要，可将handler执行改为线程池异步 |

### 4.2 风险 R2：过渡性数据存储

| 属性 | 说明 |
|:---|:---|
| **风险等级** | 🟡 中等 |
| **影响的属性** | 可靠性 |
| **风险描述** | Repository包含内存实现（ConcurrentHashMap），适用于开发测试。切换到MySQL生产环境后，事务管理、连接池、慢SQL等问题会暴露 |
| **触发条件** | 从开发环境切换到生产环境 |
| **缓解措施** | 1. 依赖倒置已生效：Domain层定义的Repository接口无需修改<br/>2. Infrastructure层已包含MyBatis映射代码<br/>3. 应用层已预留事务注解位置 |

### 4.3 风险 R3：事件日志长期增长

| 属性 | 说明 |
|:---|:---|
| **风险等级** | 🟢 低 |
| **影响的属性** | 存储和查询性能 |
| **风险描述** | event_log表记录每次操作事件，随系统运行持续增长 |
| **触发条件** | 系统运行超过1年，日志达百万级 |
| **缓解措施** | 1. 主要查询字段promotion_id已建索引<br/>2. 可设置归档策略（超1年日志迁至冷存储）<br/>3. 可按时间分区 |

## 5. 无风险决策

> **无风险决策**：基于已有证据可以确信满足质量属性目标的决策。

### 5.1 无风险 N1：Transition Table 状态机引擎（AD-02）

| 属性 | 说明 |
|:---|:---|
| **确信理由** | Table-Driven状态机是成熟模式，静态Map存储转换规则，结构清晰易验证 |
| **满足的质量场景** | MO-01（新增状态≤3处）、MO-02（新增审核节点≤3处） |
| **验证方式** | 新增一个状态 = 新增1行Map映射；新增一个事件 = 在已有Map Entry中新增1行映射 |

### 5.2 无风险 N2：三层权限控制（AD-07）

| 属性 | 说明 |
|:---|:---|
| **确信理由** | 三层权限是Web应用经典安全模式：路由守卫全局统一注册，操作权限通过纯函数computed按状态组合计算 |
| **满足的质量场景** | SE-01（拦截率100%）、SE-02（越权0%）、US-02（误操作率0%） |
| **验证方式** | beforeEach全局守卫覆盖所有路由切换；操作层computed穷举所有合法状态组合 |

### 5.3 无风险 N3：密码加密存储

| 属性 | 说明 |
|:---|:---|
| **确信理由** | BCrypt单向哈希是业界公认的密码存储标准，不可逆 |
| **满足的质量场景** | SE-03（密码加密） |
| **验证方式** | 数据库存储的是BCrypt哈希值，无法反推明文 |

### 5.4 无风险 N4：DDD领域隔离（AD-01）

| 属性 | 说明 |
|:---|:---|
| **确信理由** | 五个子领域以独立包组织，领域间无直接代码依赖。仅活动域与审核域通过LinkageValidator存在有限的协调依赖 |
| **满足的质量场景** | MO-03（SKU变更不影响其他域）、MO-04（新增角色不影响业务逻辑） |
| **验证方式** | SKU领域的代码修改不会触发活动或审核领域的编译错误 |

### 5.5 无风险 N5：事件日志完整审计

| 属性 | 说明 |
|:---|:---|
| **确信理由** | 每个事件记录操作前后的完整状态快照 |
| **满足的质量场景** | N-RE-01（操作日志支持审计，历史操作100%可查） |
| **验证方式** | 按promotion_id查询event_log表，按时间排序即可还原完整生命周期 |

## 6. 选择该架构的原因

### 6.1 业务特征驱动的选择

系统的核心业务特征是**促销活动全生命周期中的双流程联动管理**。这个特征决定了架构选择需要解决以下问题：

| 问题 | 架构如何应对 |
|:---|:---|
| 活动状态和审核状态相互影响，如何保证一致性？ | 双状态机引擎独立维护各自状态，联动校验器统一协调二者关系 |
| 管理员、审核员、定时任务三种触发源，如何统一处理？ | 事件驱动将所有操作统一为"产生事件→发布→状态机响应"的流程 |
| 未来可能增加状态或调整流程，如何控制改动范围？ | Table驱动使状态规则集中在静态Map中；DDD领域边界使模块独立演进 |
| 所有操作需要事后审计，如何记录？ | 事件驱动天然记录每次操作——事件本身即审计记录，无需额外开发 |

### 6.2 质量属性场景的满足

回顾效用树中的质量场景，逐项对照架构的支撑能力：

| 质量属性 | 关键场景 | 架构如何满足 |
|:---|:---|:---|
| 可用性 | 按钮按状态动态显隐 | 三层权限控制的操作层按状态组合计算合法操作，不合法按钮不渲染 |
| 性能 | API响应≤500ms | 状态机引擎的核心操作是Map查找+字段赋值，耗时<1ms |
| 安全性 | 角色越权成功率0% | 路由守卫拦截页面访问 + 操作层computed拦截按钮操作 |
| 安全性 | 密码不可明文 | BCrypt单向哈希存储 |
| 可修改性 | 新增状态改动≤3处 | Table驱动：改枚举（1处）+ 改Transition Table（1~2处） |
| 可修改性 | SKU变更不影响其他域 | DDD包结构隔离，领域间无代码依赖 |

### 6.3 复杂度与收益的对等

架构设计中每引入一项复杂度，必须有对应的收益：

| 引入的复杂度 | 换取的收益 |
|:---|:---|
| 四层分层（Interfaces→Application→Domain→Infrastructure） | 业务逻辑与基础设施解耦，领域层独立可测，数据库可替换 |
| 11种事件类型（而非最简的5种） | 每种操作有独立的事件类型，审计追溯到操作级别 |
| 双引擎+联动校验器（3个组件而非1个） | O(n×m)降为O(n+m)，两个状态机各自独立修改测试 |
| Pinia集中状态（而非组件内自行管理） | 6个页面共享状态，审核后自动刷新列表，无需手动emit |
| 三层权限（路由+组件+操作） | 从未登录到操作按钮的三层逐级收窄防护 |

这些复杂度的引入量与系统业务复杂度（双状态联动、三种角色、三种触发源、36种状态组合）是匹配的。

### 6.4 扩展场景的验证

以四种扩展场景验证架构的可修改性：

| 扩展场景 | 改动位置 | 改动量 | 目标 |
|:---|:---|:---|:---|
| 新增活动状态（如"暂停"） | 枚举 + Transition Table | 2处 | ≤3处 ✅ |
| 新增审核节点（如"复审"） | 枚举 + Transition Table + 联动校验器 | 2~3处 | ≤3处 ✅ |
| SKU增加字段（如"库存"） | SKU领域的Entity + Repository | 仅SKU域 | 不影响其他域 ✅ |
| 新增角色（如"超级管理员"） | User实体 + 路由配置 + usePermission | 3~4处 | ≤4处 ✅ |

### 6.5 风险评估小结

| 风险 | 等级 | 状态 |
|:---|:---|:---|
| 内存同步事件总线高并发瓶颈 | 🟡 中 | 有明确缓解路径和扩展预留 |
| 过渡性数据存储 | 🟡 中 | 依赖倒置已就绪，切换成本低 |
| 事件日志长期增长 | 🟢 低 | 可归档、可分区 |

三个风险均为中低等级，无阻塞性风险，均有对应的缓解措施。

### 6.6 总结

选择DDD分层+事件驱动的组合架构，是业务特征驱动的结果而非技术偏好。双状态机联动的业务本质决定了需要独立的状态机引擎和集中的联动协调机制；事件驱动统一了三种触发源的处理方式，同时天然满足审计追溯需求；DDD分层为复杂业务逻辑提供了清晰的组织框架。引入的架构复杂度与业务复杂度相匹配，关键质量场景均有明确的架构支撑，风险可控且有缓解路径。
